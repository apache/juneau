17  package org.apache.juneau.http.header;
18  
19  import java.util.function.*;
20  
21  import org.apache.juneau.http.annotation.*;

23  /**
24   * Represents a parsed <l>Upgrade</l> HTTP request header.
25   *
26   * <p>
27   * Ask the client to upgrade to another protocol.
28   *
29   * <h5 class='figure'>Example</h5>
30   * <p class='bcode'>
31   * 	Upgrade: HTTP/2.0, HTTPS/1.3, IRC/6.9, RTA/x11, websocket
32   * </p>
33   *
34   * <h5 class='topic'>RFC2616 Specification</h5>
35   *
36   * The Upgrade general-header allows the client to specify what additional communication protocols it supports and
37   * would like to use if the server finds it appropriate to switch protocols.
38   * The server MUST use the Upgrade header field within a 101 (Switching Protocols) response to indicate which
39   * protocol(s) are being switched.
40   *
41   * <p class='bcode'>
42   * 	Upgrade        = "Upgrade" ":" 1#product
43   * </p>
44   *
45   * <p>
46   * For example,
47   * <p class='bcode'>
48   * 	Upgrade: HTTP/2.0, SHTTP/1.3, IRC/6.9, RTA/x11
49   * </p>
50   *
51   * <p>
52   * The Upgrade header field is intended to provide a simple mechanism for transition from HTTP/1.1 to some other,
53   * incompatible protocol.
54   * It does so by allowing the client to advertise its desire to use another protocol, such as a later version of HTTP
55   * with a higher major version number, even though the current request has been made using HTTP/1.1.
56   * This eases the difficult transition between incompatible protocols by allowing the client to initiate a request in
57   * the more commonly supported protocol while indicating to the server that it would like to use a "better" protocol if
58   * available (where "better" is determined by the server, possibly according to the nature of the method and/or resource
59   * being requested).
60   *
61   * <p>
62   * The Upgrade header field only applies to switching application-layer protocols upon the existing transport-layer
63   * connection.
64   * Upgrade cannot be used to insist on a protocol change; its acceptance and use by the server is optional.
65   * The capabilities and nature of the application-layer communication after the protocol change is entirely dependent
66   * upon the new protocol chosen, although the first action after changing the protocol MUST be a response to the initial
67   * HTTP request containing the Upgrade header field.
68   *
69   * <p>
70   * The Upgrade header field only applies to the immediate connection.
71   * Therefore, the upgrade keyword MUST be supplied within a Connection header field (section 14.10) whenever Upgrade is
72   * present in an HTTP/1.1 message.
73   *
74   * <p>
75   * The Upgrade header field cannot be used to indicate a switch to a protocol on a different connection.
76   * For that purpose, it is more appropriate to use a 301, 302, 303, or 305 redirection response.
77   *
78   * <p>
79   * This specification only defines the protocol name "HTTP" for use by the family of Hypertext Transfer Protocols, as
80   * defined by the HTTP version rules of section 3.1 and future updates to this specification.
81   * Any token can be used as a protocol name; however, it will only be useful if both the client and server associate
82   * the name with the same protocol.
83   *
84   * <h5 class='section'>See Also:</h5><ul>
85   * 	<li class='link'><a class="doclink" href="https://juneau.apache.org/docs/topics/JuneauRestCommonBasics">juneau-rest-common Basics</a>
86   * 	<li class='extlink'><a class="doclink" href="https://www.w3.org/Protocols/rfc2616/rfc2616.html">Hypertext Transfer Protocol -- HTTP/1.1</a>
87   * </ul>
88   *
89   * @serial exclude
90   */
