This guide defines how Juneau documentation should be authored and published so AI agents can consume it reliably.

## Canonical Sources

Use these source-of-truth locations:

- **Concepts and behavior**: `pages/topics/*.md`
- **Versioned changes**: `pages/release-notes/*.md` and migration guides
- **API semantics**: Java sources and Javadocs in the main `juneau` repository

Do not treat `todo/*.md` as authoritative runtime behavior documentation unless content is explicitly promoted into docs/release notes.

## Canonical AI Artifacts

Generated artifacts live in:

- `static/ai/manifest.json`
- `static/ai/juneau-knowledge.jsonl`
- `static/ai/taxonomy.json`

These files are the machine-first entrypoint for AI indexing/RAG pipelines and local coding agents.

## Field Guidance (`summary` vs `description`)

Use a short `summary` for compact, token-efficient explanations and a fuller `description` for nuance:

- **`summary`**: One sentence, plain language, no paragraphs.
- **`description`**: Longer explanation with context, caveats, and examples.
- Prefer explicit summary metadata from annotations (`summary` / `su`) when available.

## Regeneration Workflow

From `juneau-docs`:

```bash
python3 scripts/generate-ai-artifacts.py
python3 scripts/check-ai-artifacts.py
```

The first command regenerates deterministic artifacts. The second verifies:

- generated AI files are up to date,
- required manifest/record shape is present,
- README quickstart version matches current Juneau release,
- Javadocs link paths in docs config are consistent,
- canonical release-note source for current release exists.

## Definition Of Done (Docs-Affecting PRs)

- New or changed feature docs are added under canonical source locations.
- AI artifacts are regenerated if source content changed.
- `check-ai-artifacts.py` passes locally.
- `summary` text is concise and `description` adds non-redundant detail.
- Source provenance remains traceable via `source_path` in JSONL output.
