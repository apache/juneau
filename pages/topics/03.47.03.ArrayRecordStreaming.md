The `parseArrayRecords(...)` / `serializeArrayRecords(...)` factories — declared by the
<a href="/site/apidocs/org/apache/juneau/marshall/stream/ArrayRecordReadable.html" target="_blank">ArrayRecordReadable</a> /
<a href="/site/apidocs/org/apache/juneau/marshall/stream/ArrayRecordWritable.html" target="_blank">ArrayRecordWritable</a>
roles — yield a multi-record cursor over the **elements of a top-level wire array**, without
materializing the whole list. This applies to single-rooted formats whose root value happens to be
a list — JSON's `[...]`, CBOR's array major-type, etc.

For naturally-multi-record formats (csv, sse), use `parseRecords` / `serializeRecords`
directly — the wire format IS already a sequence of records. JSONL is also natively
line-delimited; its `parseArrayRecords` / `serializeArrayRecords` simply alias that line
record stream (rather than wrapping the elements in a bracketed `[...]` array).

> See the <a href="/docs/topics/TokenStreamingBasics#per-format-capability-matrix">per-format capability matrix</a>
> for which formats support array-record reading/writing and whether each streams O(1) or buffers.

## Reading large arrays element-by-element

```java
// Wire payload is one big JSON array; we want O(1) memory per element.
try (RecordReader r = JsonParser.DEFAULT.parseArrayRecords(input)) {
    while (r.canRead()) {
        Bean b = r.read(Bean.class);
        process(b);
    }
}
```

## Writing large arrays element-by-element

```java
try (RecordWriter w = JsonSerializer.DEFAULT.serializeArrayRecords(output)) {
    for (Bean b : sourceOfBeans()) {
        w.write(b);
    }
}
// output is now: [{...},{...},{...}]
```

## Format applicability

| Format | Reader | Writer | Notes |
|---|---|---|---|
| JSON / JSON5 | streaming | streaming | O(1) memory per element. |
| CBOR | streaming | streaming | Reader uses indefinite-length array peek (BREAK). Writer always emits indefinite-length so no element count needed. |
| MsgPack | streaming | buffered; streaming with count overload | Reader streams O(1) per element. Writer's no-count `serializeArrayRecords(output)` buffers; `serializeArrayRecords(output, expectedCount)` writes the length-prefix header up front and serializes each element directly. |
| BSON | buffered | buffered | BSON requires byte-length prefix at every nesting level; streaming without buffering isn't possible. |
| Parquet / YAML | buffered | buffered | |
| XML / HTML | buffered with caller-specified root | not supported (reader only) | Reader has the `parseArrayRecords(input, "rootElementName")` overload; there is no array-record writer. |
| JSONL | streaming | streaming | `arrayRecords` aliases the JSONL line record stream — each top-level line is one element, written/read without a surrounding `[...]`. O(1) memory per element. |
| CSV / SSE | n/a | n/a | Use `parseRecords`/`serializeRecords` &mdash; format is already row-streamed. |
| TOML / HOCON / Hjson / INI / PlainText / Markdown / JsonSchema / JCS / Prototext | not supported | not supported | |

## Length-prefixed writers (MsgPack count overload)

For binary formats whose array wire encoding starts with a length prefix, the count overload
streams without buffering — the caller pre-declares the element count:

```java
ByteArrayOutputStream baos = new ByteArrayOutputStream();
try (RecordWriter w = MsgPackSerializer.DEFAULT.serializeArrayRecords(baos, 3)) {
    w.write(bean1);
    w.write(bean2);
    w.write(bean3);
}
// Bytes equal to: MsgPackSerializer.DEFAULT.serialize(List.of(bean1, bean2, bean3))
```

Mismatch between the declared count and the actual write count throws on `close()`. Writing
a 4th element when 3 were declared throws `IllegalStateException` immediately.

## Capability check

```java
boolean supported = MsgPackSerializer.DEFAULT instanceof ArrayRecordWritable;   // true
if (MsgPackSerializer.DEFAULT instanceof ArrayRecordWritable w)
    boolean streaming = w.isArrayRecordStreaming();   // false — buffered without the count overload
```
