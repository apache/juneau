Security is an ongoing concern in any library. Juneau is designed so that its serializers and parsers
are safe by default — POJO types are inferred from the `Class` objects you pass in rather than from
attacker-controlled input — but a few areas warrant care when you extend the framework or expose it
over HTTP.

This section covers the security characteristics of each layer:

- **[Marshalling Security](MarshallingSecurity)** — why the parsers are not subject to arbitrary-object
  (demarshalling) attacks, how bean dictionaries replace open-ended `class` attributes, and the security
  posture of the dependent parsing libraries.

- **[SVL Security](SvlSecurity)** — how the Simple Variable Language guards against variable-injection
  and file-disclosure attacks via the `Var.allowNested()` / `Var.allowRecurse()` controls, and what to
  watch for when writing your own variables.

- **[REST Security](RestSecurity)** — mitigating denial-of-service attacks through the `maxInput()`
  setting and the parsers' streaming (non-DOM) design.

- **[Content Security Policy](ContentSecurityPolicy)** — locking down Juneau's auto-generated HTML pages
  (HtmlDoc, Swagger UI, OpenAPI UI, widgets) with a first-class
  <a href="/site/apidocs/org/apache/juneau/http/header/ContentSecurityPolicy.html" target="_blank">Content-Security-Policy</a>
  header bean.

If you discover a security vulnerability in Juneau, please follow the responsible-disclosure
instructions in [SECURITY.md](https://github.com/apache/juneau/blob/master/SECURITY.md) rather than
opening a public issue.
