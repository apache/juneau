Juneau microservices are **self-contained, deployable REST servers** built on top of the Juneau REST
server framework. A microservice packages your REST resources, configuration, and an embedded HTTP container
into a single executable jar (or WAR) that you can start with a plain `java -jar` command.

The microservice layer builds on top of the full Juneau stack:

<tree>
<node-0>Juneau Microservice Stack</node-0>
<node-1>juneau-microservice — core lifecycle, config, console, inject bootstrap</node-1>
<node-1>juneau-rest-server — @Rest / @RestOp REST engine (serialization, routing, guards, …)</node-1>
<node-1>juneau-marshall — multi-format POJO marshalling (JSON, XML, HTML, YAML, …)</node-1>
<node-1>Deployment target (one of Jetty / Tomcat / Spring Boot)</node-1>
</tree>

## Deployment Targets

Juneau provides three independently-releasable deployment adapters.  All three share the same
`Microservice` + `@Configuration` + `@Bean Servlet` programming model and the same lifecycle contracts.

| Module | When to use |
|---|---|
| [juneau-microservice-jetty](/docs/topics/JuneauMicroserviceJetty) | Default choice for standalone runnable jars. Ships a zero-config `JettyMicroservice.run(args, servlet)` one-liner and a classpath-driven `jetty.xml` knob for advanced connector tuning. |
| [juneau-microservice-tomcat](/docs/topics/JuneauMicroserviceTomcat) | Drop-in alternative when your organisation standardises on Tomcat, or when you want to share `[Tomcat]` config between an embedded dev launcher and a full Tomcat deployment. |
| [juneau-rest-server-springboot](/docs/topics/JuneauRestServerSpringboot) | Integration point for Spring Boot applications. REST resources extend `BasicSpringRestServlet` / `BasicSpringRestServletGroup` so Spring `@Autowired` injection and `@Bean` resolution flow through the Spring application context instead of the Juneau bean store. |

All three adapters depend on [juneau-microservice](/docs/topics/JuneauMicroservice) for the core
lifecycle engine and on [juneau-rest-server](/docs/topics/JuneauRestServer) for the REST framework
itself.

## Lifecycle

Every Juneau microservice follows a builder → configure → start → join/stop lifecycle.

<tree>
<node-0><java-class><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.html" target="_blank">Microservice</a></java-class></node-0>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.html#create()" target="_blank">Microservice.create()</a></java-method> — obtain a fluent builder</node-1>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.Builder.html#args(java.lang.String[])" target="_blank">builder.args(args)</a></java-method> — pass CLI arguments</node-1>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.Builder.html#configurations(java.lang.Class[])" target="_blank">builder.configurations(…)</a></java-method> — register <c>@Configuration</c> classes (adds the HTTP container)</node-1>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.Builder.html#build()" target="_blank">builder.build()</a></java-method> — construct the <c>Microservice</c> instance</node-1>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.html#start()" target="_blank">start()</a></java-method> — start all <c>MicroserviceListener</c>s (including the HTTP container)</node-1>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.html#startConsole()" target="_blank">startConsole()</a></java-method> — (optional) launch interactive stdin console</node-1>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.html#join()" target="_blank">join()</a></java-method> — block the calling thread until <c>stop()</c> is called</node-1>
<node-1><java-method><a href="/site/apidocs/org/apache/juneau/microservice/Microservice.html#stop()" target="_blank">stop()</a></java-method> — stop all listeners in reverse order; close the bean store</node-1>
</tree>

All methods on `Microservice` return `this`, making it easy to chain the whole bootstrap in one
fluent expression.

## Hello World (Jetty)

The following is the smallest complete Juneau microservice: a single REST resource mounted on an
embedded Jetty server with automatic JSON/XML/HTML content negotiation.

Add the dependency:

```xml
<dependency>
    <groupId>org.apache.juneau</groupId>
    <artifactId>juneau-microservice-jetty</artifactId>
    <version>${juneau.version}</version>
</dependency>
```

Define a REST resource:

```java
import org.apache.juneau.rest.annotation.*;
import org.apache.juneau.rest.server.servlet.*;

@Rest(
    path = "/hello",
    title = "Hello World"
)
public class HelloResource extends BasicRestServlet {

    @RestGet(path = "/*", summary = "Returns a greeting.")
    public String sayHello() {
        return "Hello, World!";
    }
}
```

Boot it from `main`:

```java
import org.apache.juneau.microservice.jetty.*;

public class App {
    public static void main(String[] args) throws Exception {
        JettyMicroservice.run(args, new HelloResource()).join();
    }
}
```

`JettyMicroservice.run(args, servlet)` is a convenience facade that wires `JettyConfiguration` into
the `Microservice` builder, registers the servlet, starts on port `10000` (or as configured in
`juneau.cfg`), launches the interactive console, and blocks.  Point a browser at
`http://localhost:10000/hello` — the default HTML view renders automatically.

For the Tomcat or Spring Boot equivalents, see the module-specific pages listed in the
[Deployment Targets](#deployment-targets) section above.

## Configuration Files

Every microservice reads its runtime settings from an INI-style `juneau.cfg` file that is resolved
from the classpath or working directory at startup.  Key sections used by the built-in modules:

```ini
[Microservice]
# Name and version shown in the console banner.
name = MyApp
version = 1.0

[Jetty]
# Comma-separated list of candidate ports; first available wins.  0 = random.
port = 8080, 0

[Logging]
# Root logging level (java.util.logging).
level = WARNING
# Log file relative to the working directory.
logFile = logs/microservice.%g.log
```

The configuration is exposed as a
<a href="/site/apidocs/org/apache/juneau/config/Config.html" target="_blank">Config</a>
object through `Microservice.getConfig()` and is also registered in the
[bean store](/docs/topics/InjectAwareMicroservice) so `@Bean` factory methods can inject it directly.

### Hot-reload

`Config` supports **file-watch hot-reload** out of the box.  When the underlying `juneau.cfg` file
changes on disk, listeners registered via
<a href="/site/apidocs/org/apache/juneau/config/Config.html#addListener(org.apache.juneau.config.event.ConfigEventListener)" target="_blank">Config.addListener(…)</a>
are notified with the full set of changed entries.  The microservice uses this to propagate logging
level changes and port-list updates without a restart.

You can register your own listener from a `@Configuration` class:

```java
import org.apache.juneau.commons.inject.*;
import org.apache.juneau.config.*;
import org.apache.juneau.microservice.*;

@Configuration
public class AppConfig {

    @Bean
    MicroserviceListener configWatcher(Config config) {
        return new BasicMicroserviceListener() {
            @Override
            public void onConfigChange(Microservice ms, Config c) {
                System.out.println("Config changed: " + c.getSectionKeys("MyApp"));
            }
        };
    }
}
```

The `onConfigChange` callback fires on the calling thread of the file-store watcher, not the
request thread, so keep it non-blocking.

## Built-in Management Resources

The Jetty and Tomcat launchers expose several built-in REST endpoints under the root path when
management mixins are enabled.  These are opt-in: add the appropriate mixin to your root
`@Rest` resource.

| Endpoint | Mixin / feature | Description |
|---|---|---|
| `GET /healthz` | [Health Probes](/docs/topics/HealthProbes) | Aggregated health/liveness check. Returns `200 OK` or `503 Service Unavailable`. |
| `GET /readyz` | [Health Probes](/docs/topics/HealthProbes) | Readiness probe, used by Kubernetes load-balancer pre-stop hooks. |
| `GET /livez` | [Health Probes](/docs/topics/HealthProbes) | Liveness probe for pod restart decisions. |
| `GET /actuator` | [Management Surface](/docs/topics/ManagementSurface) | Actuator-style endpoint listing all available management sub-paths. |
| `GET /api` | Auto-generated via [Swagger / OpenAPI](/docs/topics/Swagger) | Swagger 2.x JSON schema; `OPTIONS` on any resource returns an inline schema. |
| `GET /stats` | [Execution Statistics](/docs/topics/ExecutionStatistics) | Per-operation call counts, error rates, and latency percentiles. |

The management surface is implemented via the standard Juneau
[mixin pack](/docs/topics/RestServerComposition) mechanism — there are no special microservice APIs.
The same mixins work in a plain servlet container too.

## Bean Store and Inject Bootstrap

`Microservice` owns an internal
<a href="/site/apidocs/org/apache/juneau/commons/inject/WritableBeanStore.html" target="_blank">WritableBeanStore</a>
populated from every `@Configuration` class registered on the builder.  This makes it easy for
companion modules — such as `JettyConfiguration` and `TomcatConfiguration` — to contribute lifecycle
components and for user code to wire in application services without subclassing.

```java
@Configuration
public class AppConfig {

    @Bean
    Servlet rootResources() { return new RootResources(); }   // auto-mounted by JettyConfiguration

    @Bean
    MyService myService() { return new MyService(); }         // available to other @Bean methods
}

public class App {
    public static void main(String[] args) throws Exception {
        Microservice.create()
            .args(args)
            .configurations(JettyConfiguration.class, AppConfig.class)
            .build()
            .start()
            .startConsole()
            .join();
    }
}
```

`JettyConfiguration` walks the bean store at startup, finds every `@Bean` whose declared return type
is `Servlet` and whose runtime class carries `@Rest`, and mounts it at the path declared in
`@Rest(path = "...")`.  `TomcatConfiguration` follows an identical contract.

See [Inject-Aware Microservice](/docs/topics/InjectAwareMicroservice) for a complete reference on
resolution priority, `@PostConstruct` / `@PreDestroy` callbacks, and external bean store integration.
