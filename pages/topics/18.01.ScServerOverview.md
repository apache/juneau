### Contents/Features

`juneau-sc-server` is the server component of the [juneau-sc](/docs/topics/JuneauSc) configuration
server. It is a **standalone Juneau microservice** built on `juneau-microservice-jetty` that:

- Clones or pulls a remote Git repository containing `.cfg` files.
- Exposes those files over a REST API at `GET /configs/{project}/{branch}`.
- Returns a JSON map containing up to two configuration layers: a shared `APPLICATION` default and a
  `PROJECT`-specific override.

The server depends on [JGit](https://www.eclipse.org/jgit/) for Git operations and on
`juneau-rest-server` / `juneau-microservice-jetty` for the REST layer.

## Running the Server

The server is launched as a standard executable jar:

```java
public class App {

    public static void main(String[] args) throws Exception {
        Microservice
            .create()
            .args(args)
            .configurations(JettyConfiguration.class, AppConfig.class)
            .build()
            .start()
            .startConsole()
            .join();
    }

    @Configuration
    public static class AppConfig {

        @Bean
        public Servlet loadConfigResource() {
            return new LoadConfigResource();
        }
    }
}
```

The `LoadConfigResource` servlet is automatically mounted by Jetty at the path declared by its
`@Rest(path="/configs/*")` annotation.

## Server Configuration File

The server reads its own settings from a `juneau-server-config.cfg` file located in the working
directory. The most important section is `[GitServer]`:

```ini
[Jetty]
# Port(s) to listen on — first available wins; 0 = random
port = 10000,0,0,0

[GitServer]
# Local filesystem path where the repo will be cloned
pathLocal = /var/lib/juneau-sc/config-repo

# Remote Git repository URI
gitRemote = https://github.com/my-org/my-config-repo.git
```

On each request the server:
1. Checks whether `pathLocal` exists.
   - If not, it runs a `git clone` of `gitRemote` into that directory.
   - If yes, it runs a `git pull`.
2. Checks out the requested branch (`git checkout origin/{branch}`).
3. Reads `application.cfg` (shared defaults) and `{project}.cfg` (project override) from the local clone.

## REST Endpoint

```
GET /configs/{project}/{branch}
```

| Segment | Description |
|---------|-------------|
| `{project}` | Name of the project; the server looks for `{project}.cfg` in the repo root. |
| `{branch}` | Git branch to check out before reading files. |

**Success response** — `200 OK`, `Content-Type: application/json`:

```json
{
    "APPLICATION": {
        "value": "<raw text of application.cfg>"
    },
    "PROJECT": {
        "value": "<raw text of {project}.cfg>"
    }
}
```

If `application.cfg` is absent, the `APPLICATION` key is omitted.  
If `{project}.cfg` is absent, the `PROJECT` key is omitted.

**Example:**

```bash
curl http://localhost:10000/configs/my-service/main
```

## Security Considerations

The current implementation does not perform authentication or authorisation on the `/configs` endpoint.
Before exposing it outside a trusted network, consider:

- Adding HTTP Basic Auth or OAuth via a Juneau `@RestGuard` or servlet filter.
- Restricting access at the network layer (firewall rules, service mesh policy).
- Storing Git credentials outside the config file (environment variables or a secrets manager)
  rather than hard-coding them.

The `GitControl` class currently uses a `UsernamePasswordCredentialsProvider` with credentials stored
in the source. This should be externalised to environment variables or a secrets store before production
deployment.
